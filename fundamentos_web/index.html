    Aplicações Web e Arquitetura de Software¶

    Diferentes Camadas na Arquitetura de Software¶

    Uma camada é como uma separação lógica e física dos componentes em uma aplicação ou serviço. Esta separação é feita em nível de componentes, não em nível de código.

    Aplicações em camada única¶

    Em aplicações de camada única a interface do usuário, o backend da lógica do negócio e o banco de dados estão na mesma máquina.

    flowchart BT
    A[(Database)] <--> B[Server] <--> C[User Interface]

    Aplicações Desktop são exemplos de aplicações camada única.

    Vantagens de Aplicações Camada Única¶

    • Não há latência de conexão, pois cada componente está na mesma máquina.
    • Aplicações multicamada mandam requisições frequentes ao servidor, o que adiciona latência e reduz a velocidade no sistema, piorando a experiência do usuário.
    • Privacidade e segurança: Os dados do usuário permanecem sempre na mesma máquina e não precisam ser transmitidos por rede.

    Desvantagens Aplicações Camada Única¶

    • O publicador da aplicação não tem controle sobre a aplicação, uma vez que o software é entregue, não podem ser realizadas atualizações ou manutenção, a menos que o cliente manualmente baixe um patch de instalação.
    • O código é vulnerável para ser mexido ou alvo de engenharia reversa, pode ser modificado ou copiado por alguém mal intencionado.
    • A performance da aplicação depende da máquina de quem executa.

    Aplicações Camada Dupla¶

    Uma aplicação camada dupla envolve um cliente e um servidor. O cliente contém a interface do usuário com a lógica do negócio em uma máquina. Enquanto o servidor backend inclui um banco de dados rodando em uma máquina diferente. O banco de dados é hospedado pelo negócio e têm o controle sobre ele.

    flowchart LR
    A[Cliente]--"Requisição"-----> B[Servidor Backend]
    B[Servidor Backend]--"Resposta"-----> A

    Exemplos de utilidade: Aplicativos de produtividade, jogos no browser. Os dados pesados são baixados uma vez e o servidor é requisitado apenas para salvar o estado.

    Aplicações Três Camadas¶

    Esse tipo de aplicação é bastante popular e amplamente utilizado na internet. Quase todo website simples como blogs e jornais fazem parte dessa categoria.

    Em aplicações Três Camada, a interface do usuário, a lógica do negócio e o banco de dados, todos estão em máquinas diferentes, estão separadas fisicamente.

    flowchart LR
    subgraph Cliente
    direction LR
    a1[cliente 1]
    a2[cliente 2]
    a5[cliente 3]
    end
    cliente <--> a3
    a3[Servidor] <--> a4[(Banco de Dados)]

    Aplicações N-Camadas¶

    Uma aplicação que tem mais de 3 componentes envolvidos na arquitetura.

    Componentes podem ser:

    • Cachê.
    • Mensageria.
    • Load Balancers.
    • Servidores de busca.
    • Componentes para processamento de grandes quantidades de dados.
    • Componentes rodando serviços web, micro serviços.

    Popularmente conhecidos como Sistemas distribuídos.

    Dois princípios do desenvolvimento de software são as chaves para justificar esta arquitetura: Responsabilidade única e Separação de Conceitos.

    Diferença entre Layers e Tiers: Layers são em níve lde código (conceitual/lógico), enquanto tier está relacionado à separação física dos componentes.

    Arquitetura Web¶

    Arquitetura Cliente-Servidor¶

    Essa arquitetura é peça fundamental da internet.

    Funciona no modelo requisição-resposta. O cliente manda a requisição da informação para o servidor e o servidor responde com a informação.

    Cliente¶

    O cliente detém a interface do usuário. A interface do usuário é a parte da aplicação da apresentação. É escrita em HTML, JavaScript, CSS.

    Cliente Magro¶

    O cliente detém apenas a interface do usuário da aplicação. Não contém nenhum tipo de lógica. Para toda ação o cliente manda um requisição para o servidor de backend, como em uma aplicação de 3 camadas.

    Cliente Gordo¶

    Detém toda ou alguma parte da lógica do negócio. Estas, são Aplicações Duas Camadas.

    Servidor¶

    A tarefa primária de um servidor é receber requisições do cliente e fornecer a resposta depois de executar a lógica do negócio baseado nos parâmetros da requisição recebidos pelo cliente.

    Todo serviço online precisa de um servidor para funcionar, servidores rodando aplicações web são conhecidos como servidores de aplicação.

    Renderização do lado do servidor¶

    Os desenvolvedores usam um servidor para renderizar a interface do usuário no backend e então mandam os dados gerados para o cliente.

    Comunicação entre Cliente e Servidor¶

    Protocolo HTTP¶

    Toda a comunicação da web acontece através do protocolo HTTP. HTTP é um protocolo requisição-resposta que define como a informação é transmitida pela internet.

    É um protocolo sem estado, e todo processo sobre HTTP é executado independentemente e não tem conhecimento dos processos anteriores.

    REST API e Extremidades API¶

    No contexto de aplicações modernas multicamadas , todo cliente tem que atingir uma extremidade REST para buscar os dados no backend.

    O código da aplicação no backend tem uma API REST implementada. Isso age como interface para as requisições do mundo externo. Toda requisição, seja ela do cliente, escrita pelo negócio ou por desenvolvedores terceiros, os que consomem a API precisam atingir as extremidades REST para buscar os dados.

    API REST¶

    REST significa Representational State Transfer. É um estilo de arquitetura de software para implementar web serviços. Serviços implementados utilizando o estilo REST são conhecidos como web serviços RESTful.

    Uma API REST é uma implementação de API que adere às restrições do estilo REST, ela age como uma interface. A comunicação entre cliente e servidor acontece através de HTTP, a API REST tira vantagem das metodologias HTTP disso para estabelecer comunicação entre o cliente e o servidor. REST também habilita servidores para cachear a resposta que melhora a performance da aplicação.

    A comunicação entre cliente e servidor é livre de estado, ou seja, toda comunicação entre cliente e servidor é como uma nova comunicação.

    REST endpoint¶

    Um endpoint API/REST/Backend significa uma URL do serviço que o cliente pode alcançar.

    API Gateway¶

    A API REST age como um gateway ou um único ponto de entrada no sistema. Ela encapsula a lógica do negócio e lida com todas as requisições dos cliente, tomando de conta das tarefas de autorização, autenticação, sanitização dos dados de entrada, antes de prover acesso para os recursos da aplicação.

    HTTP Push e Pull¶

    HTTP Pull - Polling With AJAX¶

    Existem duas formas de buscar os dados do servidor.

    A primeira é mandar uma requisição HTTP GET para o servidor manualmente, ativando um evento na interface do usuário ao clicar em um botão ou interagindo com outro elemento da página.

    A outra é buscar dinamicamente em intervalos regulares usando AJAX sem qualquer intervenção humana.

    AJAX significa Asynchronous JavaScript e XML. Ajax é usado para adicionar comportamento assíncrono à página web.

    AJAX usa um objeto XHTMLHttpRequest para mandar requisições ao servidor. Este objeto vem com o browser e usa JavaScript para atualizar o HTML DOM.

    AJAX é comumente usado com o framework jQuery para implementar o comportamento assíncrono na UI.

    Essa técnica dinâmica de requisitar informação do servidor em intervalos regulares é conhecida como polling.

    HTTP PUSH¶

    Time to Live (TTL)¶

    Na comunicação regular entre cliente e servidor, que é o HTTP PULL, existe um TTL para cada requisição, pode ser de 30 a 60 segundos, variando de acordo com o browser.

    Se o cliente não receber resposta dentro do TTL, o browser mata a conexão e o cliente tem que reenviar a requisição.

    Conexões abertas consomem recursos e existe um número de conexões abertas que um servidor pode lidar, se as conexões não fecham e novas são introduzidas o servidor ficará sem memória.

    Conexão Persistente¶

    Se certa resposta leva mais tempo do que o TTL do browser, precisamos de uma conexão persistente entre o cliente e o servidor.

    Uma conexão persistente é uma conexão de rede entre o cliente e o servidor que permanece aberta para requisições e respostas futuras, ao invés de serem fechadas após uma única comunicação.

    • Heartbeat interceptors: Requisições em branco entre o cliente e o servidor para prevenir que o browser mate a conexão.
    • Uso intensivo de recursos para manter a conexão aberta, deve ser usado quando for vital para o funcionamento da aplicação.
    • Implementação: AJAX Long Polling, Web Sockets, Server-Sent Events, etc.
    Tecnologias HTTP baseadas em PUSH¶
    Web Sockets¶

    Uma conexão web socket é preferível quando precisamos de um fluxo de dados bi-direcional de baixa latência do cliente para o servidor e voltando.

    Casos típicos: aplicações de chat, streams sociais de tempo real, MMO browser e etc. Estes aplicativos têm um número significante de leituras e escritas comparados a apps web normais.

    Com web sockets, podemos manter conexão cliente servidor aberta o tempo que quisermos.

    Web sockets não funcionam com HTTP, eles funcionam com TCP. Ambos os servidores e clientes devem suportar web sockets para que funcione.

    AJAX - Long polling¶

    Long polling está em algum lugar entre AJAX e Web Sockets. Nesta técnica, ao invés de retornar uma resposta vazia automaticamente, o servidor mantém a respostas até que encontre uma atualização para ser enviada ao cliente.

    A conexão no long polling permanece a aberta um pouco mais de tempo comparado ao polling. Se a conexão quebrar, o cliente deve reestabelecer a conexão ao servidor.

    HTML5 Event-Source API e Server-Sent Events¶

    Ao invés do cliente buscar pelos dados, o servidor automaticamente empurra os dados para o cliente sempre que uma atualização esta disponível. As mensagens advindas do servidor são tratadas como eventos.

    Nessa abordagem, os servidores podem iniciar a transmissão dos dados para os clientes uma vez que o cliente estabelece a conexão.

    Isso ajuda a eliminar um número considerável de ciclos requisição-respota em branco, cortando o consumo de banda.

    Para implementar server-sent events, a linguagem de backend deve suportar a tecnologia. Na interface do usuário, HTML5 Event-Source API é usada para receber dados vindos do backend.

    Uma vez que o cliente estabelece a conexão, o fluxo de dados é unidirecional apenas, do servidor para o cliente.

    Streaming over HTTP¶

    Ideal para casos em que precisamos stremar muitos dados através de HTTP quebrando em pequenos pedaços. Iisso é possível com HTML5 uma API JavaScript Stream.

    Renderização Cliente vs Servidor¶

    Quando o browser recebe uma página de web de um servidor como resposta, ele tem que renderizar a resposta na janela na forma de uma página HTML.

    Para fazer isso o browser tem vários componentes, tais como:

    • Engine do browser
    • Engine de renderização
    • Interpretador JavaScript
    • Backend de Rede e UI
    • Armazenamento de dados e etc.

    A engine de renderização constrói a árvore DOM e renderiza a construção.

    Renderização no Servidor¶

    Para cortar esse tempo de renderização, os desenvolvedores geralmente renderizam a UI no servidor, geram o HTML lá e direcionam a página HTML para a UI.

    Não é compatível com AJAX, pois necessitaria reenviar completamente a página atualizada para o cliente.

    Podemos utilizar uma abordagem híbrida, renderizando a parte estática no servidor e a parte dinâmica no cliente.

    Escalabilidade¶

    Escalabilidade é o potencial de uma aplicação de lidar com uma crescente carga de trabalho sem sacrificar a performance.

    Latência¶

    É o tempo que o sistema leva para responder à uma requisição de um usuário. Sistemas almejam possuir latência mínima.

    A latência é medida como a diferença de tempo em que o usuário toma uma ação e a resposta do sistema.

    A latência é usualmente dividida em duas partes: A latência da conexão e a latência da aplicação.

    Latência da Conexão¶

    É o tempo em que a rede leva para mandar um pacote de dados do ponto A para o ponto B. A rede deve ser eficiente o suficiente para lidar com o aumento do tráfego.

    Para diminuir a latência, negócios usam uma CDN (Content Delivery Network) para implementar seus servidores ao redor do globo o mais próximo possível do usuário final. Essas localizações próximas ao usuários são conhecidas como Edge Locations.

    Latência da aplicação¶

    É o tempo que a aplicação levar para processar uma requisição do usuário. Existem várias formas de diminuir a latência da aplicação. O primeiro passo é executar testes de stress e de carga. na aplicação para detectar os gargalos que afetam o sistema.

    Tipos de escalabilidade¶

    Para escalar uma aplicação é necessário um sólido poder computacional. Os servidores devem ser potentes o suficiente para lidar com o aumento do tráfego.

    Existem duas formas de escalar uma aplicação: Escalamento Horizontal e Vertical.

    Escalabilidade Vertical: Aumentar o poder computacional do servidor. A escala vertical possui limites óbvios.

    Escalabilidade Horizontal: Aumentar a quantidade de servidores. A escala horizontal permite escalamento dinâmico em tempo real quando necessário.

    A mais importante razão porque computação em nuvem têm se tornado mainstream na indústria é devido a habilidade da nuvem de escalar dinamicamente.

    O processo de adicionar e remover servidores, aumentando e voltando à capacidade original é chamado de Elasticidade da Nuvem.

    O código necessário para rodar sistemas distribuídos deve ser livre de estado. Não devem haver instâncias estáticas no código, instâncias estáticas mantém os dados da aplicação no servidor particular, quando o servidor cai, os dados/estado é perdido.

    Uma boa prática é evitar o uso de instâncias estáticas de uma classe. O estado é tipicamente persistido em uma memória distribuída, isso facilita que os componentes sejam livre de estado.

    É por isso que programação funcional se tornou popular para sistemas distribuídos, as funções não retém nenhum estado.

    Possíveis Gargalos¶

    • Banco de Dados
    • Projeto da aplicação
    • Não uso de caching
    • Lógica de negócio no banco de dados
    • Uso do banco de dados errado
    • Código ineficiente e mal escrito

    Alta Disponibilidade¶

    Alta disponibilidade é o modelo atual na indústria. É a habilidade de um sistema de permanecer online apesar da ocorrência de falhas no nível da estrutura e tempo real.

    Um serviço altamente disponível significa que o sistema estará online em 99.99999% do tempo. Ou seja, terá um nível de serviço de 99.99999%.

    O nível de serviço é um fator crítico para alguns serviços , como sistemas de aviação, sistemas espaciais, máquinas de mineração, servidores de hospitais, mercado de capitais.

    Para obter essa disponibilidade o sistema deve ser projetados para serem tolerante a faltas e seus componentes são construídos de forma redundante.

    Razões para falhas de sistemas¶

    • Crash de software
    • Falhas de hardware
    • Erros Humanos
    • Desligamento programado

    Arquitetura Monolítica¶

    Uma aplicação tem um arquitetura monolítica se ela contém toda a aplicação em uma única base de código. Uma aplicação monolítica é autocontida.

    Monolitos são mais simples de construir, testar e implementar em comparação à arquitetura de micro-serviços.

    Vantagens da Arquitetura Monolítica¶

    Simplicidade: são simples de desenvolver, testar, implementar, monitorar e gerenciar, visto que tudo está em apenas um repositório.

    Contras da Arquitetura Monolítica¶

    Implementação contínua é uma dor em monolitos, até mesmo uma pequena mudança de código pode resultar na necessidade de uma reimplementação completa.

    Necessidade de Testes regressivos, visto que uma mudança de código pode impactar potencialmente a funcionalidade de outras features do app.

    Um Único Ponto de Falha pode derrubar toda a aplicação.

    Manutenção e Escalabilidade são um desafio em monolitos.

    Dificuldade em usar tecnologias diferentes.

    Arquitetura Micro Serviços¶

    Na arquitetura de Micro Serviços, diferentes features de um serviço extensivo são implementadas separadamente como serviços menores vagamente atrelados chamados micro serviços. Estes micro serviços trabalham em conjunção para formar um grande serviço distribuído.

    Cada serviço tem uma única responsabilidade de executar uma feature específica e é separado dos outros serviços. Esta arquitetura facilita uma manutenção mais rápida e limpa, o desenvolvimento de novas features, o teste e a implementação de módulos individuais.

    Bancos de Dados¶

    Um banco de dados é um componente na arquitetura de uma aplicação necessário para persistir os dados.

    Dados podem ser: Estruturados, Não Estruturados, Semi-Estruturados e dados de estado do usuário.

    Bancos de Dados Relacionais¶

    Bancos de Dados Relacionais persistem dados contendo relacionamentos: one to one, one to many, many to many, many to one, etc. É o tipo de banco de dados mais usado em desenvolvimento web.

    Bancos de Dados Relacionais possuem um modelo de dados relacional, dados são organizados em tabelas contendo linhas e colunas e SQL é a linguagem de consulta primária para interagir com dados relacionais.

    Consistência dos dados¶

    Bancos relacionais, asseguram a gravação dos dados de forma normalizada. Uma entidade pode ocorrer apenas em um lugar/tabela.

    Isso ajuda a manter a consistência nos dados, podemos atualizar os dados em apenas um lugar ao invés de atualizar o dado em múltiplas tabelas.

    Transações ACID¶

    Bancos de Dados Relacionais asseguram transações ACID.

    ACID significa atomicidade, consistência, isolamento e durabilidade.

    Uma transação ACID significa que se uma transação ocorre, ela será executada com perfeição sem afetar qualquer outro processo ou transação. Depois que a transação é completada o sistema terá um novo estado que será durável e consistente. No caso de alguma coisa errada acontecer durante a transação, como uma pequena falha no sistema, toda a operação será revertida.

    Um transação ACID acontece com um estado inicial do sistema, estado A, e se completa com um estado final do sistema, estado B. Ambos estes estados são duráveis e consistentes.

    Bancos de Dados NO SQL¶

    Construídos para leituras e escritas de alta frequência.

    • Escalável
    • Roda em Clusteres
    • Mais fáceis de aprender e amigáveis ao desenvolvedor
    • Sem SQL
    • Inconsistência
    • Sem suporte para ACID

    Diferentes Tipos de Banco de Dados¶

    • BD orientado a documentos
    • Data store Chave-valor
    • Wide-Column DB
    • BD Relacional
    • BD em Grafo
    • BD série temporal
    • DB dedicado a aplicativos mobile...

    Caching¶

    Caching é a chave para a performance de qualquer aplicação, assegura baixa latência e alta taxa de transferência. Uma aplicação com caching sempre fará melhor do que uma aplicação sem caching, simplesmente porque um cachê intercepta todas as requisições para o banco de dados em fornece a resposta instantaneamente.

    Implementar caching é uma aplicação web significa copiar dados frequentemente acessados do banco de dados, que é baseado em hardware de disco, e armazenar na RAM para resposta rápida.

    Caching pode ser usados para dados dinâmicos (com tempo de sobrevivência - TTL) ou dados estáticos.

    Fila de Mensagens¶

    Uma fila de mensagem é uma fila que faz a rota de mensagem da fonte para o destino, ou do servidor para remetente para o receptor seguindo a política FIFO (First In, First Out).

    Além de FIFO, filas de mensagens também suportam delivery baseado em prioridade, mensagens têm uma prioridade atribuída a elas e a fila processa as mensagens baseado no conjunto de prioridades. Essas filas são chamadas de filas de prioridade.

    Características da Fila de Mensagens¶

    Filas de mensagens facilitam o comportamento assíncrono, esse comportamento permite que os módulos se comuniquem sem dificultar suas atividades primárias.

    Também facilitam a comunicação entres os módulos, o que é chave para uma arquitetura orientada a serviços e micro serviços, elas permitem a comunicação em um ambiente heterogêneo, fornecendo armazenamento temporário para mensagens até que elas sejam processadas e consumidas pelo consumidor.
