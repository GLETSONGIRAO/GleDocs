    Engenharia de Software¶

    Práticas de Engenharia de Software¶

    • Escrever código limpo e modular
    • Melhorar a eficiência do código
    • Adicionar documentação efetiva
    • Usar controle de versão
    • Teste
    • Logging
    • Code Reviews

    Código de Produção¶

    Código em produção precisa ser confiável e eficiente.

    Código Limpo: Legível, simples e conciso. Uma característica do código em produção essencial para colaboração e manutenibilidade do software.

    Modular: Logicamente dividido em funções e módulos. Torna o código mais organizado, eficiente e reusável.

    Um módulo é um arquivo que permite que o código seja reusado. Encapsulando o código em arquivos que podem ser importados em outros arquivos.

    Refatoração¶

    Reestruturação do código para melhorar a estrutura interna sem mudar a funcionalidade externa.

    A refatoração nos dá a chance de escrever de escrever código limpo e modular depois que ele já está funcionando.

    Vantagens:

    • Reduz a carga de trabalho a longo prazo.
    • Mias fácil de dar manutenção no código.
    • Mais fácil de reusar o código.
    • Ajuda a tornar um melhor desenvolvedor.

    Namespaces¶

    • Usar nomes descritivos e implicar o tipo.
      • Ex: is_ ou has_ para booleanos, age_list para listas.
    • Ser consistente, mas diferenciar claramente.
    • Evitar abreviações e letras únicas.
      • Com exceção de contadores como i ou j.
    • Evitar nomes longos, colocar apenas informação relevante nos nomes.

    Uso de espaço em branco¶

    • O padrão de identação em python são 4 espaços.
    • Separa seções com espaço em branco.
    • Limitar as linhas em 79 caracteres, docstrings em 72 caracteres.

    Escrevendo código Modular¶

    • DRY (Don't Repeat Yourself).
    • Abstrair a lógica para melhorar a legibilidade do código.
    • Uma função deve fazer apenas uma coisa.
    • Tentar usar menos do que 3 argumentos por função.

    Eficiência do Código¶

    • Código Eficiente

      • Executa mais rápido.

      • Ocupa Menos espaço na memória.

    • Usar operações vetorizadas (NumPy).

    • Teste de pertencimento em Sets é mais rápido do que em listas.

    Documentação¶

    Texto adicionar ou ilustração que vem com o Software ou está embutido no código.

    Útil para clarificar partes complexas do código, tornando o código mais fácil de navegar e entender porque diferentes componentes do programa são usados.

    Diferentes tipos de documentação podem ser usados em diferentes níveis do programa:

    • Comentários in-line.
    • Docstrings.
    • Documentação do projeto.

    Teste¶

    Testar o código é essencial antes da implantação, teste ajuda a descobrir erros e conclusões precipitadas antes que elas tenham um impacto maior.

    Test Driven Development: Um processo de desenvolvimento onde teste são escritos para uma tarefa antes mesmo do código para a execução da tarefa.

    Teste Unitário: Um tipo de teste que cobre uma "unidade" de código, geralmente uma única função, de forma independente do resto do programa.

    Testes unitários tem a vantagem de serem isolados do resto do programa e não ter dependências envolvidas, não necessitam de acessos a banco de dados ou APIs. Entretanto, para garantir que o programa inteiro esteja funcionando corretamente é necessário fazer testes de integração.

    Ferramentas para Testes Unitários: Começar com Pytest.

    Logging¶

    Logging é o processo de gravar mensagens que descrevem os eventos que ocorreram enquanto o programa está rodando.

    Logging é útil para entender os eventos que ocorrem enquanto o problema executa.

    Escolher o nível apropriado para logging:

    • DEBUG - Para qualquer coisa que acontece no programa.
    • ERROR - Gravar qualquer erro que ocorre.
    • INFO - Gravar todas as ações ocasionadas pelo usuário ou específicas do sistema.

    Dica: Fornecer qualquer informação útil.

    Code Review¶

    As perguntas abaixo ajudam a conduzir uma code review.

    O código é limpo e modular?

    • Pode ser entendido facilmente?
    • Nomes e espaços são significantes?
    • Pode ser fornecida outra camada de abstração?
    • Cada módulo é necessário?
    • Algum módulo ou função é muito longa?

    O código é eficiente?

    • Existe algum loop que pode ser vetorizado?
    • Podem ser aplicadas melhores estruturas de dados?
    • Pode ser diminuída a quantidade de cálculos necessários em cada passo?
    • Podem ser usados geradores ou multiprocessamento para otimizar algum passo?

    A documentação é efetiva?

    • Os comentários in-line são concisos e significativos?
    • Existe algum código complexo que a documentação não explica?
    • As funções têm docstrings efetivas?
    • A documentação necessária para o projeto é fornecida?

    O código está bem testado?

    • O código tem boa cobertura de testes?
    • Os testes checam para casos interessantes?
    • Os testes são legíveis?
    • Testes podem ser mais eficientes?

    O logging é efetivo?

    • As mensagens de log são limpas, concisas e profissionais?
    • Elas incluem toda a informação útil e relevante?
    • Elas usam o nível apropriado de logging?

    Programação Orientada a Objetos (OOP)¶

    Programação Orientada a Objetos tem algumas vantagens sobre programação procedural:

    • Permite a criação de programas grandes, modulares que podem ser facilmente expandidos.
    • Esconde as implementações do usuário final.

    Objetos são definidos por características e ações.

    Definições¶

    • Classe: Uma receita, consistindo de métodos e atributos.
    • Objeto: Um instância de uma classe.
    • Atributos: Uma descrição ou característica.
    • Método: Uma ação que um objeto pode tomar.
    • Encapsulamento: Uma das ideias fundamentais da OOP. Podemos combinar funções e dados em um única entidade. Permite esconder detalhes da implementação.

    Métodos Get e Set¶

    Em OOP tradicional, acesso e mudança de atributos são usadas com os métodos Get e Set, e os atributos são definidos com _attribute como forma de avisar que um atributos é privado, mesmo isso não existindo em python.

    class Shirt:

    def __init__(self, shirt_color, shirt_size, shirt_style, shirt_price):
        self._price = shirt_price

    def get_price(self):
      return self._price

    def set_price(self, new_price):
      self._price = new_price

    Docstrings¶

    Não é necessário definir "self" na docstring do método, já está subtendido que qualquer método terá self como primeiro input.

    class Pants:
    """The Pants class represents an article of clothing sold in a store
    """

    def __init__(self, color, waist_size, length, price):
        """Method for initializing a Pants object

        Args: 
            color (str)
            waist_size (int)
            length (int)
            price (float)

        Attributes:
            color (str): color of a pants object
            waist_size (str): waist size of a pants object
            length (str): length of a pants object
            price (float): price of a pants object
        """

        self.color = color
        self.waist_size = waist_size
        self.length = length
        self.price = price

    def change_price(self, new_price):
        """The change_price method changes the price attribute of a pants object

        Args: 
            new_price (float): the new price of the pants object

        Returns: None

        """
        self.price = new_price

    def discount(self, percentage):
        """The discount method outputs a discounted price of a pants object

        Args:
            percentage (float): a decimal representing the amount to discount

        Returns:
            float: the discounted price
        """
        return self.price * (1 - percentage)
